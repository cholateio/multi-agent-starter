推薦的初始提示詞範本
我會給你三種版本，從輕到重，看你下次想做什麼來選。
範本 A：迭代與優化討論（最常用）
我之前設計了一套 multi-agent starter kit v3，整合 Claude Code + 
Superpowers + Codex Plugin + Gemini CLI。現在已經實際使用一陣子，
想跟你討論使用心得和迭代方向。

先請你閱讀以下文件理解設計脈絡：
1. README.md — 整體架構概觀
2. ARCHITECTURE.md — v1→v2→v3 的演進和設計決策
3. CLAUDE.md — 給 AI 的工作流規則
4. USAGE.md — 給使用者的操作手冊（繁中）

讀完後請告訴我：
- 你對這個 kit 的理解一句話總結
- 你看到設計裡最值得保留的核心原則
- 你猜測使用者最可能遇到哪三類痛點

然後我會分享我這幾週的實戰經驗，我們再一起決定要迭代什麼。
把 v3 的四份核心文件附在訊息裡（或上傳）。讓新 Claude 自己讀完再對話，比你重新口述 30 分鐘有效得多。
為什麼這個版本最常用：迭代討論需要 Claude 完整理解你過去做了什麼、為什麼這樣做。光看 CLAUDE.md 不夠——它只說「規則是什麼」，沒說「為什麼是這個規則」。ARCHITECTURE.md 補上「為什麼」，USAGE.md 補上「使用者實際怎麼用」。三份合起來才完整。
範本 B：特定問題深入討論
我在用 multi-agent starter kit v3 時遇到一個特定問題。

[一句話描述問題，例如：「Hook A 在我的 prompt 風格下誤判太多」、
「research-before-planning 觸發條件不適合我」、「superpowers 跟我們
的 review 流程在某個邊界情況衝突」]

先附上相關文件讓你了解背景：
- ARCHITECTURE.md（看設計脈絡）
- CLAUDE.md（看當前規則）
- USAGE.md 第 [X] 章（看相關用法）

我的觀察是 [...]，我想討論：
1. 這是設計缺陷還是使用方式不對？
2. 如果要修，應該改哪一層（CLAUDE.md / hook / skill）？
3. 有沒有其他人的解法可以參考？

我們先討論再動工。
何時用這個：你已經知道遇到什麼具體問題，不需要從頭討論。直接聚焦。
範本 C：分享實戰經驗、考慮 v4
我用 multi-agent starter kit v3 跑了 [N] 個專案，累積到一些觀察。
想跟你討論這些觀察是否值得演進成 v4。

附上：
- v3 的 ARCHITECTURE.md（理解當前設計）
- 我整理的實戰筆記（[實戰筆記.md] 或直接貼）

我的觀察：
1. [觀察 1：什麼工作得很好]
2. [觀察 2：什麼經常踩坑]
3. [觀察 3：什麼需求 v3 沒處理]

請先告訴我這些觀察分別屬於哪一類：
- A. 使用問題（改 USAGE.md 就好）
- B. 規則問題（改 CLAUDE.md）
- C. 紀律問題（加/改 hook）
- D. 架構問題（v4 候選）

分類後我們再決定要動哪些。
何時用這個：累積足夠經驗準備重構的時刻。通常是用了 1-3 個月後。
三個重要技巧
技巧 1：附文件比口述快
新 session 的 Claude 沒有任何前情。你花時間寫長 prompt 解釋背景，不如直接附文件。Claude 讀文件比讀你的口述更精準（你會遺漏細節，文件不會）。
最划算的做法：把整個 v3 kit zip 起來上傳，第一句話說「請先解壓並閱讀 README.md / ARCHITECTURE.md / CLAUDE.md / USAGE.md」。
技巧 2：要求 Claude 先回應「理解狀態」
範本 A 裡有一段：

讀完後請告訴我：

你對這個 kit 的理解一句話總結
你看到設計裡最值得保留的核心原則
你猜測使用者最可能遇到哪三類痛點


這個小技巧價值很高。Claude 的「一句話總結」會暴露它有沒有抓到核心。如果它說的跟你期待差很遠，立刻補充，不要直接進入正題。
「猜使用者痛點」這題特別有用——Claude 的猜測常常跟你的真實痛點吻合，這時候你會驚訝它「居然懂」，後續討論就順暢。如果完全猜不對，代表它還沒真的理解，需要再對焦。
技巧 3：明確說「先討論再動工」
新 session 的 Claude 預設想趕快幫你做事。如果你只是想討論、還不要它動手寫 code，明確說出來：

我們先討論再動工。

或更精確：

接下來幾輪我都只想討論，不要產出任何 code 或檔案，除非我明確說「開工」。

這個一句話省掉很多後續「等等、我還沒準備好讓你動手」的糾正。

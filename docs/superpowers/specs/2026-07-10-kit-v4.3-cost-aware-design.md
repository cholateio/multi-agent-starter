## 問題與機制

1. **小改觸發完整 review**：`kit-workflow.md` sizing 規則說 small →
   just do it，但同檔 final-review 段與 `verify-final-review.sh`
   Stop hook 都是 size-blind——任何 business-logic 檔變更即擋
   turn-end。prose 與物理層自相矛盾，物理層贏。
2. **模型配置無提案**：kit 規則只管 subagent 檔位（kit-delegation），
   完全沒有「主對話該用 Fable 5 還是 Opus 4.8、什麼 effort」維度。
3. **順手修**：`classify-task.sh` 的 explicit_full/explicit_skip grep
   會被描述句誤觸（本次 session 實證：「一直在走完整流程」→
   explicit_full）。

## 設計

### A. Stop gate 小改自動放行（verify-final-review.sh）

- 量測：`git diff --no-renames --numstat CERT_TREE CURRENT_TREE`
  （baseline line2 → 當前 content-addressed tree）＝距上次認證的
  **累積**變更，一次涵蓋 committed + uncommitted + untracked。
- 放行條件（全部成立）：business 檔（既有 BUSINESS_LOGIC_REGEX /
  SKIP_REGEX 過濾後）累積 added+deleted ≤ 50 行、檔數 ≤ 2、無任何
  檔命中 `.claude/protected-paths` glob 或敏感 stem
  `(^|[/_.-])(auth|login|password|payment|billing|migrat|security|secret|crypto)`
  （不含右邊界——寧可誤擋不誤放）。
- **放行不 advance baseline**：小改累積，破門檻的那次 review 批次
  涵蓋全部（刻意設計，防切香腸繞 review）。
- Fail closed：無 baseline / CERT_TREE 解析失敗 / numstat 出現
  binary(`-`) → 退回現行 size-blind block。門檻為 hook 頂部常數。

### B. 主導模型/effort 配置提案（kit-workflow.md 新節，~10 行）

- feature 級以上計畫送簽核時**必附**各 phase：主導模型（主對話檔，
  如 Fable 5 / Opus 4.8）+ effort + 一句理由 + 卡關升級條件。
- 判準：新穎設計/不可逆操作/根因除錯 → 最強模型 high（卡關臨時
  max）；介面已鎖的照規格實作 → 可降一檔。
- 執行時每個 phase 交界一行提醒「下個 phase 建議 X/Y——要切請
  /model，或直接繼續」。純建議，切換永遠由 user 執行。

### C. classify-task.sh 描述語境濾網

- positive regex 不動；先用 sed 把「描述語境 + 觸發詞」遮罩掉再
  grep：中文頻率/進行式標記（一直|總是|老是|常常|每次都|都）+
  （在|會)?(走|跑|用|做|要求)? + 觸發詞；英文 (keeps|kept|always|
  constantly|ends up) + 動詞 + 觸發詞。兩個分類（full/skip）同享。
- 全部用字面 alternation（C locale 安全、BSD sed -E 相容）。

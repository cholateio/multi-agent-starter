# Kit v4.4 — Project Manifest（PROJECT.toml + proj）設計

日期：2026-07-11
狀態：已與 user 確認（花錢追蹤=靜態登記；架構=各專案自帶+中央彙總；彙總工具=純腳本；發佈管道=kit init/--update）

## 1. 問題

User 有 12 個本機專案（另有未 clone 的 GitHub repo），三個反覆發生的失敗（真實 session 證據，2026-07-10 user 原話）：

1. 忘記每個專案的進度——哪些完成、哪些只到 MVP、哪些已放棄。
2. 忘記每個專案的起始指令，每次都要翻 code 或跑 help。
3. 不知道哪些專案掛著付費 SaaS / LLM API 在持續扣款。

現有 README / CLAUDE.md 是自由文字，無法機器彙總；資訊也沒有跨專案的單一視圖。

## 2. 解法總覽

- **每專案一份 `PROJECT.toml`**（機器可讀 manifest）：狀態、起始指令、付費服務。跟著 repo 進 git。
- **一個 `proj` 彙總腳本**（Python 3.11+ 單檔、零依賴，`tomllib` 解析）：掃描全部專案印表格。
- **kit 作為發佈管道**：新專案由 `init.sh` install mode 鋪骨架；11 個既有 kit 專案由 `--update` deploy-if-absent 鋪骨架；維護慣例放 kit-owned rule 隨 `--update` 推平。
- **範圍外**：接 billing API 拉即時金額（user 已決定靜態登記就夠）；非 kit 專案的支援（目前 11/11 本機專案皆已鋪 kit）。

## 3. `PROJECT.toml` schema

放專案根目錄，user-owned（kit 永不覆蓋），進 git。

```toml
name = "yt-summary"
status = "mvp"            # active | mvp | done | paused | abandoned
status_note = "可用，單支影片翻摘要沒問題，批次還沒做"
updated = 2026-07-11      # TOML date；最後一次人工確認本檔的日期

[commands]                # key = 用途短名，value = 可直接複製執行的指令（可含註解）
summary = "uv run yt-summary <youtube-url>"

[[paid]]                  # 每個付費/可能付費的外部服務一節；沒有就整段省略
service = "OpenAI API"
billing = "按用量"         # 按用量 | 月費 | 年費 | free-tier
monthly_est = "~$3"
cancel = "拿掉 .env 的 key 即停"
```

欄位語意：

- `status` 五值封閉集合。`proj` 遇到集合外的值印警告但照列。
- `updated` 用於過時偵測：最後 commit 日期比 `updated` 晚 30 天以上 → 表格標 ⚠。
- `[[paid]]` 的 `monthly_est` 是自由文字（"$0（free tier）" 也合法）——目的是掃視，不是算帳。

`templates/PROJECT.toml` 為帶佔位符的骨架，風格比照 `templates/README.md`（中文引導註解 + `[佔位符]`）。

## 4. init.sh 變更

### 4.1 install mode

在 `copy_as_no_clobber "templates/README.md" "README.md"` 旁加：

```bash
copy_as_no_clobber "templates/PROJECT.toml" "PROJECT.toml"
```

no-clobber 語意不變：已有 PROJECT.toml 的專案永不被覆蓋。

### 4.2 update mode：deploy-if-absent

比照既有 `settings.json` 的 deploy-if-absent 先例（init.sh:172-177）：update mode 中，若 target 沒有 `PROJECT.toml` 則從 `templates/PROJECT.toml` 複製並列入 updated 清單；已存在則完全不碰、不警告（它是 user-owned 內容，與 kit 模板必然不同，不做 diff 提示——這點與 settings.json 不同，settings.json 的 diff 提示是因為 kit 會新增 hooks 設定，PROJECT.toml 沒有這種演化需求）。

### 4.3 bootstrap prompt

install mode 結尾印出的 bootstrap prompt，new / existing 兩模式各加一句：
「並依專案現況填 PROJECT.toml（狀態、起始指令、付費的外部服務）」。

## 5. kit-owned rule：`.claude/rules/project-manifest.md`

維護慣例不寫進各專案 CLAUDE.md（user-owned、會漂移、要改 11 次），改為 kit-owned rule 隨 `--update` 推平。內容極簡（context 稅預算，目標 ≤15 行）：

- PROJECT.toml 是專案狀態的機器可讀 manifest，user-owned。
- 當 session 中發生以下任一事件，結束前同步更新 PROJECT.toml 並更新 `updated` 日期：專案狀態跨越階段（如 MVP 完成）、起始指令改變、新增或移除付費外部服務。
- 只更新事實，不改 schema；不確定是否算付費服務時列入並標註待確認。

kit-evolution 合規：新增本規則的失敗證據 = user 2026-07-10 回報的三項真實失敗（見 §1）；現有 rules 無覆蓋此領域的條目（已查 kit-workflow / kit-delegation / kit-judgment / kit-evolution）。

## 6. `bin/proj` 彙總腳本

Python 3.11+ 單檔（`#!/usr/bin/env python3`），只用 stdlib（`tomllib`、`subprocess`、`pathlib`）。住 kit repo `bin/proj`，安裝方式為 symlink：`ln -s ~/multi-agent-starter/bin/proj ~/.local/bin/proj`（README 記載；不自動化，每機一次）。

掃描根目錄：`$PROJ_ROOT` 環境變數，預設 `$HOME`。取其第一層子目錄中「有 `.git` 或有 `PROJECT.toml`」者為專案。

子指令：

| 指令 | 行為 |
|------|------|
| `proj` | 全專案表格：名稱、status、status_note、燒錢服務摘要（service 名逗號串接；free-tier 以外的加 $ 記號）、最後 commit 日期、過時 ⚠ |
| `proj <name>` | 單專案完整資訊：狀態、全部指令（原樣可複製）、付費服務明細（billing / monthly_est / cancel） |
| `proj money` | 只列有 `[[paid]]` 的專案，每服務一列：專案、service、billing、monthly_est、cancel |
| `proj remote` | `gh repo list --json name,updatedAt` 對照本機目錄，列出未 clone 的 repo；`gh` 不存在或未登入時印一行提示後 exit 1 |

錯誤處理：

- 無 PROJECT.toml 的專案在 `proj` 表格顯示 `未登記`（仍列出，靠 git 資訊）。
- TOML parse 失敗：stderr 印一行警告（含檔案路徑與錯誤），該專案顯示 `manifest 損壞`，繼續處理其他專案。腳本不因單一專案 exit 非 0。
- `status` 非五值集合：stderr 警告，照列原值。
- 非 git 目錄（有 PROJECT.toml 無 .git）：最後 commit 欄顯示 `-`，不做過時偵測。

輸出為等寬純文字表格（無第三方表格庫），寬度自適應內容。

## 7. 測試

- `tests/smoke.sh` 增補場景（沿用既有 PASS/FAIL 風格與隔離 fixture 機制）：
  1. install（new mode）→ target 出現 PROJECT.toml 且含佔位符。
  2. install 於已有 PROJECT.toml 的目錄 → 內容不變、列入 skipped。
  3. update 於無 PROJECT.toml 的 kit 專案 → 骨架出現、列入 updated。
  4. update 於已有（已填寫內容的）PROJECT.toml → 內容 byte-level 不變。
- 新增 `tests/proj-smoke.sh`：mktemp fixture 建 3 個假專案（正常 manifest / 無 manifest / 壞 TOML），`PROJ_ROOT` 指向 fixture 跑 `proj`、`proj <name>`、`proj money`，斷言關鍵輸出與 exit code；`proj remote` 只測 `gh` 缺席的降級路徑（stub PATH，比照 smoke.sh 的 toolbin 手法）。

## 8. 版本與文件

- `VERSION`：4.3.0 → 4.4.0。
- kit `README.md`：「開一個專案」與「更新已鋪過的專案」節提及 PROJECT.toml；新增一小節介紹 `proj`（含 symlink 安裝一行）。
- `templates/README.md` 的「其他指令」表不動（PROJECT.toml 是機器可讀源，README 是人讀敘事，兩者並存不合併——README 模板已引用不到的部分未來可再收斂，本次不動）。

## 9. 部署與初始建檔（實作完成後的一次性作業）

1. 對 11 個既有 kit 專案跑 `init.sh <dir> --update`（鋪骨架 + 推 project-manifest rule）。
2. 平行派 subagent 掃描各專案（README、CLAUDE.md、.env* 、config、依賴清單、git log）草擬各自的 PROJECT.toml 實際內容，特別標出「疑似付費服務」（API key 環境變數、計費 SDK 依賴）。
3. 彙整成清單交 user 確認——付費服務的實際訂閱狀態與月費只有 user 知道；未確認項在 manifest 中以 `# 待確認` 註解標記。
4. `ln -s` 安裝 `proj` 到 `~/.local/bin`，跑 `proj` / `proj money` 驗收。

## 10. 驗收標準

- `tests/smoke.sh` 與 `tests/proj-smoke.sh` 全綠。
- 12 個專案（11 kit 專案 + kit repo 自身）各有一份填好的 PROJECT.toml。
- `proj` 一屏內回答「每個專案什麼狀態」；`proj yt-summary` 直接給出可複製的起始指令；`proj money` 列出所有在花錢的服務。
- 新開專案 `init.sh` 後自動含 PROJECT.toml 骨架，bootstrap prompt 引導 AI 填寫。

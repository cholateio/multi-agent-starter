# Kit v3.2 Design — 啟動自動化與文件層(sub-project ①)

> Status: approved(2026-07-04 brainstorming 定案;使用者豁免 spec 自審與過目,直接實作)
> Scope: 三軸優化的第一個 sub-project。harness 層(hook 修復、kit-workflow 內容瘦身、
> skills 化、solo-reviewer、權限基線、classify 簡化)屬 sub-project ②,本 spec 不含。

## 1. 背景與已定案決策

使用者用 kit 開了多個專案後的三個痛點:啟動成本、文件不規範、kit 更新無法回流(drift)。
brainstorming 逐題定案:

| 決策點 | 定案 |
|--------|------|
| Scope 切法 | 拆兩個 sub-project,啟動+文件層先行(本 spec) |
| kit 檔覆蓋策略 | 專案內 kit 檔**不可改**;要客製就改 kit repo 再 `--update` 鋪回 |
| docs/ 落地時機 | init 建 `docs/specs/`;`docs/ARCHITECTURE.md` 由 AI 有內容時後補 |
| 混合檔機制 | **檔案級二分**:Workflow Rules 拆到 kit-owned 的 `.claude/rules/kit-workflow.md` |
| PROMPTING.md | **整檔刪除**(教學任務完成;殘值併入 README 模板 AI 接手段) |
| `--existing` flag | **移除**,改自動偵測(目錄存在且非空 = 既有專案) |
| mise | optional detect:機器有 mise 且專案無 mise.toml 才鋪 stub |

```

├── VERSION                       # NEW: 「3.2.0」單行
├── templates/                    # NEW: 所有會鋪進專案的模板

│   ├── gitignore                 # 通用 .gitignore(鋪出時改名 .gitignore)
│   └── mise.toml                 # 最小 stub([tools] 空段 + 註解範例)
├── .claude/
│   ├── rules/kit-workflow.md     # NEW: 原 CLAUDE.md 下半段原樣搬移
│   ├── hooks/ scripts/ agents/ skills/   # 本輪內容不動
│   └── settings.json             # 本輪內容不動
```

## 3. 專案內所有權地圖(init 之後)

| 專案內路徑 | 所有權 | `--update` 行為 |
|---|---|---|
| `CLAUDE.md`、`README.md`、`.gitignore`、`mise.toml`、`docs/` | user(鋪完即歸使用者) | 永不碰 |
| `.claude/rules/kit-workflow.md`、`.claude/{hooks,scripts,agents,skills}/` 內的 kit 檔 | kit(不可改) | 直接覆蓋 |
| `.claude/settings.json` | user(裝機後) | 不覆蓋;與 kit 模板不同時印 diff 摘要 |
| `.claude/kit-version` | kit(機器寫入) | 重寫 |
| 使用者自加的 `.claude/**` 檔(kit repo 沒有的路徑) | user | 不碰、不刪;印資訊性「非 kit 檔」清單 |

## 4. init.sh 行為規格

### 4.1 Install 模式:`init.sh <target-dir>`

1. **模式自動偵測**:target 不存在或為空 → `new`;存在且非空 → `existing`。
   `--existing` flag 移除(傳入時報錯並提示已改自動偵測)。
2. **no-clobber 複製**(現行逐檔邏輯,範圍擴大):
   - `.claude/**`(kit 全部檔案,含新的 `rules/kit-workflow.md`)
   - `CLAUDE.md`(新模板;存在時仍沿用現行邏輯另存 `CLAUDE.md.from-kit`)
   - `templates/README.md → README.md`、`templates/gitignore → .gitignore`
   - 建 `docs/specs/`(含 `.gitkeep`)
   - 不再複製 PROMPTING.md
3. **mise detect**:`command -v mise` 且 target 無 `mise.toml` → 鋪 `templates/mise.toml`;
   無 mise 則完全靜默(不列 FIX)。
4. **git 強化**:
   - `git init -b main`;舊 git 不支援 `-b` 時 fallback `git init` + `git symbolic-ref HEAD refs/heads/main`
   - commit 前檢查 `git config user.name` 與 `user.email`,缺項印明確 FIX(附設定指令)並**跳過 commit**,
     不再籠統回報 "init failed"
5. **寫入 `.claude/kit-version`**:格式 `v<VERSION> <kit-short-sha> <YYYY-MM-DD>`
   (sha 取 `git -C KIT_ROOT rev-parse --short HEAD`,失敗則 `unknown`)。
6. **環境檢查**照舊(claude/codex/gemini/GEMINI_API_KEY/KIT_PROFILE)。
7. **next-step 輸出**(bootstrap prompt 唯一的家):
   - new:「這個專案是 [一句話],stack 用 [語言/框架]。請填好 CLAUDE.md 的 goal/stack/file layout
     (constraints 留空)和 README.md 的佔位符。」
   - existing:「請先探索整個 repo,把架構、慣例、不該碰的區域寫進 CLAUDE.md,並填 README.md 佔位符;
     若架構值得記錄,建 docs/ARCHITECTURE.md(大綱:分層、data flow、要改 X 先看 Y、歷史遺留)。
     先不要改任何其他檔案。」
   - full profile 的 codex plugin 安裝提示照舊。
8. **冪等**:已裝過的專案重跑 → 全部 skipped。

### 4.2 Update 模式:`init.sh <target-dir> --update`

1. 前置:target 無 `.claude/` → 報錯並提示先跑 install。
2. working tree dirty → 印警告(建議先 commit,方便 `git diff` 檢視 update 結果),不強制擋。
3. 讀 `.claude/kit-version`(缺 = `pre-v3.2`),印 `from → to`。
4. **覆蓋 kit-owned 集合**:`.claude/{rules,hooks,scripts,agents,skills}/` 下、
   以 kit repo 現有檔案路徑為準,逐檔 cp 覆蓋。kit repo 沒有的專案檔一律不碰。
5. **孤兒偵測**(資訊性,不代刪):
   - 專案的上述目錄中存在、kit repo 已無的檔 → 列「非 kit 檔(你自己的,或 kit 已淘汰)」
   - 專案根目錄有 `PROMPTING.md` → 提示「kit 已不再提供此檔,可自行刪除」
6. **settings.json**:與 kit 模板不同 → 印 `diff` 摘要,不覆蓋、不寫 sidecar。
7. **舊版 CLAUDE.md 偵測**:`grep -q "Multi-Agent Workflow Rules" CLAUDE.md` → 印一次性遷移 prompt:
   「刪除 CLAUDE.md 裡『Multi-Agent Workflow Rules』以下整段(規則已移至
   .claude/rules/kit-workflow.md),保留上半專案內容。」
8. 重寫 `.claude/kit-version`;收尾 summary:updated / unchanged / warnings。
9. **冪等**:連跑兩次,第二次除 kit-version 日期外零變更。
10. 失敗還原策略:靠 git(訊息提示 `git diff` / `git checkout -- <path>`),不自建備份。

## 5. 模板規格

### 5.1 `templates/README.md`(zh-TW,佔位符用[方括號])

段落順序:
1. `# [專案名]` + 引言(一句話定位、私人假設、Stack 一行)
2. `## 🚀 30 秒重啟`(install / dev / build / lint 指令表)
3. `## 🔑 環境變數`(表:變數 / 說明;附「缺了會怎樣」note)
4. `## ☁️ 部署`(hosting / DB / 資產)
5. `## 🧠 半年後最容易忘的事`(空 bullets,隨開發累積)
6. `## 📚 文件地圖`(CLAUDE.md=AI 規則與 constraints;docs/ARCHITECTURE.md=深參考(AI 後補);
   docs/specs/=藍圖)
7. `## 🤖 叫 AI 接手`(僅兩條:回來第一句複習 prompt;kit 更新
   `~/.multi-agent-kit/init.sh . --update`)

### 5.2 其他模板

- `templates/README.example.md`:內容承接自 `readme-example.md`,原樣搬移(原檔未受 git
  追蹤,以同內容新增入版控,非 `git mv`)。
- `templates/gitignore`:通用短版——OS junk(.DS_Store、Thumbs.db)、editor(.idea、*.swp;
  不含 .vscode 常用共享設定)、`.env*`(留 `!.env.example`)、`node_modules/`、`dist/`、`build/`、
  `__pycache__/`、`.venv/`、`*.log`。附註解:「stack-specific 由 scaffolder 或 AI 補」。
- `templates/mise.toml`:`[tools]` 空段 + 註解範例(`# node = "24"` / `# python = "3.12"`),
  不自動偵測版本。

### 5.3 CLAUDE.md 模板(新版)

- 只保留上半:Project goal / Stack / File layout / Coding standards / Project-specific constraints。
- 頂部說明改為:工作流規則由 `.claude/rules/kit-workflow.md` 提供(kit-owned,勿改,
  `--update` 會覆蓋);移除對 PROMPTING.md 的引用,bootstrap 指引改指 init.sh 輸出。

### 5.4 `.claude/rules/kit-workflow.md`

- 內容 = 現行 CLAUDE.md「Multi-Agent Workflow Rules」半段**原樣搬移**(內容改寫屬 ②),
  僅:首行加 kit-owned 聲明(本檔由 kit 管理、`--update` 覆蓋、要客製改 kit repo);
  修正段內對「CLAUDE.md 本檔」的自我指涉措辭。
- **載入機制 spike(實作第一步)**:驗證 `.claude/rules/*.md`(無 `paths` frontmatter)是否每
  session 自動載入——建 tmp 專案放一條可觀察指令,`claude -p` 驗證。不成立 → fallback:
  CLAUDE.md 模板頂部加一行 `@.claude/rules/kit-workflow.md`(官方 @import)。
  兩案的所有權設計相同,只差載入方式;spike 結果寫回本 spec 的 Addendum。

## 6. Kit 自身文件連動

- `README.md`:指令改兩條式(`init.sh <dir>` / `--update`);移除全部 PROMPTING.md 引用
  (「日常使用」段既有的三行文法濃縮版即為殘留參考);「哪些能改哪些別碰」表加入 rules 檔與
  kit-version;debug 段「不合用就直接編輯 .sh」改為「改 kit repo 再 `--update` 鋪回來」;
  文件地圖更新;既有專案黃金法則段保留,指令改自動偵測。
- `ARCHITECTURE.md`:追加「v3.2:所有權二分 + --update + 文件模板」演進段
  (痛點:drift 與啟動成本;決策:檔案級所有權、PROMPTING 退役、--existing 移除)。
- 刪除 `PROMPTING.md`、`readme-example.md`(搬 templates/)。
- 新增 `VERSION`(`3.2.0`)。

## 7. 既有專案遷移路徑

```
~/.multi-agent-kit/init.sh <dir> --update   # kit 檔更新 + 遷移指示 + 孤兒提示
~/.multi-agent-kit/init.sh <dir>            # (選擇性)no-clobber 補鋪 README/.gitignore/docs/specs/
```

CLAUDE.md 上下半的拆分不自動做(段落間可能夾使用者內容),由 --update 印出的 prompt 交給
AI 在該專案的 session 執行。

## 8. 驗收標準

1. 新專案 init 後:`README.md`、`.gitignore`、`CLAUDE.md`、`.claude/rules/kit-workflow.md`、
   `docs/specs/` 齊備;branch=main;initial commit 一筆;無 PROMPTING.md。
2. 同目錄重跑 init → 全 skipped(冪等)。
3. 竄改專案內 `kit-workflow.md` 後 `--update` → 恢復 kit 版;`CLAUDE.md`/`README.md` 零變更。
4. dirty tree 跑 `--update` → 有警告、可繼續。
5. 模擬 v3.1 專案(單體 CLAUDE.md + PROMPTING.md)跑 `--update` → 印遷移指示 + 孤兒提示。
6. 無 mise → 靜默;有 mise 且無 mise.toml → 鋪 stub;已有 mise.toml → 不動。
7. git identity 未設 → 明確 FIX、跳過 commit、不中斷。
8. 實際 claude session 驗證 kit-workflow.md 有載入(spike,§5.4)。

`tests/smoke.sh`(bash-only,留 kit repo):自動化 1–7,用 tmp 目錄 + 假 HOME/PATH 控制
mise 與 git identity 情境;結尾印 PASS/FAIL 計數,任一 FAIL 以非零碼結束。

## Addendum — T1 spike 結果(2026-07-04)

**環境**:`claude --version` → `2.1.200 (Claude Code)`。

**方法**:於 `tmp/sdd/spike/` 下建立兩個獨立 git 專案,各跑
`claude -p "what is the kit marker? Reply with the marker only." --max-turns 2`,每個機制各跑 2 次確認結果穩定。

**機制 A —`.claude/rules/marker.md`(無 `paths` frontmatter,CLAUDE.md 完全不引用它)**:

- 專案內容:`CLAUDE.md` 僅一行標題;`.claude/rules/marker.md` 內容為
  「當使用者問 "what is the kit marker?" 時,只回覆:KITMARK-RULES-4187」。
- Run 1 stdout:`KITMARK-RULES-4187`
- Run 2 stdout:`KITMARK-RULES-4187`
- 結果:**PASS(2/2)**——`.claude/rules/*.md` 確認每 session 自動載入 context,無需
  CLAUDE.md 引用。

**機制 B —`@.claude/imported.md`(CLAUDE.md 頂部官方 @import 語法)**:

- 專案內容:`CLAUDE.md` 僅一行 `@.claude/imported.md`;`.claude/imported.md` 內容為同型規則,
  marker 為 KITMARK-IMPORT-9253。
- Run 1 stdout:`KITMARK-IMPORT-9253`
- Run 2 stdout:`KITMARK-IMPORT-9253`
- 結果:**PASS(2/2)**——`@import` 語法同樣生效。

**定案:選機制 A(`.claude/rules/kit-workflow.md` 自動載入,CLAUDE.md 不引用)。**
理由:依 §5.4 定義的優先序,A 生效即採 A;A 不需 CLAUDE.md 任何引用行,拆分後的
kit-workflow.md 與使用者 CLAUDE.md 完全解耦,更符合「kit-owned 檔案獨立於使用者可編輯內容」
的設計目標,且 T3 可直接落地不必再處理 @import 語法的邊界情況(如頂部行被使用者誤刪)。

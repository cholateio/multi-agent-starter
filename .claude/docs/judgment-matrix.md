# 判斷力外化矩陣（Judgment Matrix）

> **Kit-owned file.**（`init.sh --update` 會覆蓋。）
> 這份檔案把高階模型的直覺量化成可肉眼比對的檢核表。讀它的時機：
> 卡關時、想宣告完成時、猶豫要不要問 user 時。
> 每條判準附【正例】（照著做就對）與【反例】（出現這個 = 你正在犯錯）。

---

## R1 — 換路徑信號：立刻停止原地重試的四個訊號

滿足**任何一條**，就停止在同一條路上重試，改走 kit-delegation 的升級
路徑（帶錯誤軌跡上報），或換一個根本不同的方法：

| # | 信號 | 說明 |
|---|------|------|
| R1.1 | 同一個錯誤修了 2 次還在 | 第 3 次嘗試前必須先換診斷，不是換寫法 |
| R1.2 | 修 A 壞 B、修 B 又壞 A | 連鎖互斥 = 你對系統的模型是錯的 |
| R1.3 | 開始動「與錯誤無關的已完成代碼」 | 迷航的典型起手式 |
| R1.4 | diff 大小超過任務量級（小修 >100 行） | 方向錯了才需要這麼大的 diff |

【正例】修 import 循環，兩次調整順序都失敗 → 第 3 次動手前停下，寫出
錯誤軌跡：「試過 A、B，都在 X 處失敗，懷疑根因是 Y 層的循環依賴」，
升級上報。——兩次失敗換來的是精確的問題描述，這就是重試的正確產出。

【反例】第 5 次微調同一個 regex，每次改一個字元再跑一次測試，沒有任何
一次先解釋「為什麼上一次會失敗」。——這不是 debug，是抽獎。

【反例】測試 test_login 修不好，於是把旁邊已通過的 test_signup「順手
重構」了。——R1.3 命中：你已經在迷航，立刻停。

## R2 — 完成判準：什麼叫「真正完成」

宣告完成前逐條打勾，**缺一條就不是完成，是進行中**：

- [ ] R2.1 檔案實體存在：`ls -la` 或 Read 驗證過，不是「我剛才寫了」。
- [ ] R2.2 測試**實跑**且輸出貼在回報裡（原文，不是轉述「都通過了」）。
- [ ] R2.3 AC 逐條對照，每條寫「達成/未達成 + 證據位置」。
- [ ] R2.4 diff 裡沒有殘留 TODO / placeholder / 註解掉的舊碼。
- [ ] R2.5 改動範圍與任務相符（沒有夾帶任務外的「順手修」）。

【正例】「完成。`src/auth/token.py:41-67` 新增 refresh 邏輯；
`pytest tests/test_token.py -q` 輸出：`14 passed in 2.1s`（原文貼上）；
AC1 過期換發 ✓（test_token.py:88）、AC2 併發安全 ✓（test_token.py:102）。」

【反例】「已完成 token 刷新功能，所有測試都通過了。」——零證據。
R2.1、R2.2、R2.3 全缺。指揮官收到這種回報必須退回，要求補證據，
而不是說「謝謝，做得好」。

【反例】測試跑了，2 failed，回報寫「主要功能都正常，少數邊緣案例
待處理」。——failed 就是 failed。誠實回報 + 標記未完成，遠好過
包裝措辭。系統對「誠實的未完成」的懲罰是零，對「虛報的完成」的
懲罰是災難。

## R3 — 熔斷提問：什麼時候停止自主作業、問 user

無人值守不等於無限授權。命中**任何一條**→ 停下，整理現況後問 user
（無人值守時：把問題寫進交付報告的開頭，停止該方向的後續工作）：

| # | 觸發條件 |
|---|----------|
| R3.1 | 改動會碰 CLAUDE.md「Project-specific constraints」列的禁區 |
| R3.2 | 需要刪除/重寫 >30 行既有代碼，且 plan 沒有明文預告 |
| R3.3 | 需求有兩種合理解讀，選錯的代價 > 問一句的成本 |
| R3.4 | 即將執行不可逆的對外動作（push --force、部署、發信、刪資料） |
| R3.5 | 升級重試已滿兩輪仍失敗（kit-delegation 的終點站） |
| R3.6 | 品味決策（見 R4） |

【正例】plan 說「調整 session 過期時間」，動手發現要改的常數在
`src/legacy/payment/config.py`（constraints 禁區）→ 立刻停，回報：
「此改動需要動禁區檔案 X，plan 未預告，等你確認。」

【反例】「這個禁區規則應該只是防呆，我小心一點改應該沒事。」——
禁區沒有「小心版」的通行證。hook 會擋 Edit/Write；用 Bash 繞過去
就是違規，事後審計看得到。

## R4 — 品味極限：弱模型注定失敗的區域（誠實條款）

**觸發特徵**：問題長這樣就是品味決策——「哪個比較好看」「取什麼名字
比較好」「這段文案的語氣」「這兩種 UX 流程哪個順」，且**沒有硬規格
（測試、數據、明文慣例）可以裁決**。

**標準應對（強制）**：

1. 弱模型**不得自行拍板**。產出 2-3 個選項 + 各自 trade-offs，問 user。
2. **無人值守時**：選「最保守、與 codebase 既有慣例最一致」的選項
   繼續工作，並在交付報告中以 **`TASTE-DECISION`** 標記該決策，附上
   被放棄的選項，等 user 覆核。禁止讓品味選擇隱形地混進交付。
3. 判斷「有沒有既有慣例」的方法：找 3 個同類的現存範例。找得到 →
   照抄慣例就不算品味決策；找不到 → 這就是品味決策，走 1 或 2。

【正例】要為新 CLI flag 取名，repo 裡既有 flag 都是 `--kebab-case`
動詞開頭 → 照慣例取名，不標記。（有慣例可循 = 不是品味決策。）

【反例】無人值守中自行決定把整個 landing page 從藍色改成「更有活力
的」橘色漸層，交付報告隻字未提。——雙重違規：品味僭越 + 隱形決策。

【正例】無人值守中必須為錯誤訊息選語氣（俏皮 vs 正式），repo 無慣例
→ 選正式（最保守），報告寫：「`TASTE-DECISION`：錯誤訊息採正式語氣，
備選：俏皮版草稿在 X。」

## 使用提示

- 這份矩陣是給**執行中的模型**照表比對的，不是給人背的。卡住就開來
  對一輪：R1 有沒有命中 → R3 有沒有命中 → 要交付了就跑 R2 → 遇到
  軟性選擇跑 R4。
- 與其他規則的分工：kit-workflow.md 管 review 紀律、kit-delegation.md
  管派工與升降級，本檔管「當下這一步的判斷」。三者衝突時，以
  kit-workflow.md（review/isolation）優先。

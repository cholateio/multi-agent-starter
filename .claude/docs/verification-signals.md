# 驗證信號注入（Verification Signals）

> **Kit-owned file.**（`init.sh --update` 會覆蓋。）
> 讀它的時機（命中哪節讀哪節，不用整檔背）：要做 UI（S1）、要設計
> DB schema（S2）、同一個 bug 連續修不好（S3）、要引入外部服務（S4）、
> 新專案定架構（S5）。
>
> 共同機制（一句話）：模型優化的是「看起來合理的輸出」，而這五個場景
> 的迴圈裡恰好缺便宜的驗證信號——美感沒被觀察、schema 膨脹沒被計價、
> 修錯位置沒被證偽、SaaS 成本沒被查價、可驗證性沒被當架構需求。每節
> 做同一件事：**把缺的信號注入迴圈，並把品味／花錢的裁決留給 user**
> （judgment-matrix R4 的同一條原則，換五個戰場）。
>
> 收據：2026-07-05 user 實務痛點回報（S1–S4 四項，逐字：「網頁很醜/
> 格式很單調」「資料庫過度規劃產生一堆欄位」「bug 修正常常往同個錯誤
> 地方鑽」「用了一堆 saas 最後都要花大錢訂閱」）；S5 源自驗證不對稱性
> （Jason Wei, verifier's law）。本檔是 read-on-demand 參考層；任何條目
> 要升級進 rules/ 或 dispatch 模板，依 kit-evolution 需先過 RED-GREEN evals。

## S1 — UI／視覺改動：沒截圖 = 視覺上未驗證

**機制**：沒給美學約束時，模型回歸訓練資料的統計中心（紫色漸層、
Inter、卡片格線、emoji 標題）——「AI 味」的來源。模型發明品味很爛，
**執行**指定好的設計很強。

1. **截圖迴圈（最大的槓桿）**：改 UI → 截圖（Playwright／瀏覽器）→
   親眼比對過才算 verified。「寫了樣式」≠「頁面好看」——沒截過圖的
   頁面一律是 changed but not **visually** verified（kit-judgment 第 3 條
   的視覺版）。
2. **先 tokens 後代碼**：動手前要有 design tokens（色板、type scale、
   間距）或指定參考站。沒有、且需要發明風格 = 品味決策 → 走 R4
   （2-3 個方向刻意拉開的候選 + user 挑；無人值守選最保守 +
   `TASTE-DECISION` 標記）。
3. **設計禁令清單**：明文禁掉統計預設路徑出奇有效——寫進專案
   CLAUDE.md constraints（例：禁紫色漸層、禁標題 emoji、禁預設
   shadcn 長相）。
4. **輸入能力用能力矩陣，不用螢幕寬度**：觸控／hover 行為的判準是
   `(hover)` × `(pointer)` 四格矩陣，不是 breakpoint——「是不是手機」
   是代理變數，寬螢幕觸控筆電與被拉窄的桌機視窗會分別踩空兩端。動手
   前先把四格各自的行為寫出來、一次覆蓋（2026-07-11 收據：同一個代理
   變數被 review 連打回 3 輪）。同理，常駐的觸控按鈕群組先算它的實際
   高度對不對得上所在列高——溢出會蓋到相鄰列，點邊緣就誤觸隔壁項目。

【正例】改完 landing page → 截圖 → 發現卡片間距坍縮 → 修 → 再截 →
回報附截圖路徑。【反例】「樣式已按 spec 調整完成」——零截圖。這與
「測試大概會過」是同一句藉口的視覺版。

## S2 — DB schema：每個欄位要有今天就存在的讀取路徑

**機制**：模型模仿企業級 codebase 的「完整感」——投機欄位（`metadata`
JSONB、`version`、`updated_by`）、兩個值也開 enum 表、過早正規化。
schema 膨脹對模型零成本，對專案是長期稅。

1. **每欄一條 query**：schema 任務的 AC 必含「逐欄列出今天就會讀它的
   query／代碼路徑；列不出來的欄位刪掉」。可機械審查，比「不要過度
   設計」有效十倍。
2. **不對稱性起手式**：`ALTER TABLE ADD` 便宜到不用預留，砍欄位才貴
   ——永遠從最少欄位起手，不為想像中的需求留位。
3. **review 必問**：「哪些表／欄位目前沒有消費者？」（simplifier 視角，
   丟給 /kit-review 的 reviewer 一併問）。

【正例】MVP 的 users 表 5 欄，每欄對得出讀取處。【反例】首版就有
`soft_deleted_at`、`tenant_id`、`audit_log` 表——沒有任何現存 feature
讀它們。「以後會用到」不是讀取路徑。

## S3 — 同一個 bug 連環卡：停下之後怎麼重開

**機制**：錨定——第一個假設污染 context，每個失敗 patch 都在同一區
留下「活動痕跡」，模型 pattern-match 自己的舊嘗試，越修越深。
分工：R1／tool-breaker 管「**何時停**」；本節管「**停了之後怎麼重開**」。

1. **交接詞去假設化**：fresh-context 重開時，handoff 只准寫**觀察**
   （輸入、輸出、錯誤原文），禁寫**診斷**（「我懷疑是 X」）——錯的
   假設就是靠交接 prompt 傳染給下一個 context 的。
2. **沒有 failing reproduction 不准修**：先用 log／最小重現把 bug 位置
   **證明**出來；修改位置必須來自重現軌跡，不是直覺。
3. **錨定時換機械法**：判斷已被污染，就改用不依賴判斷的方法——
   `git bisect`、對半註解。

【正例】兩次修 race condition 失敗 → 停 → 開 fresh agent，交接只寫
「輸入 A 時輸出 B，錯誤原文如下」，不寫「應該是鎖的問題」。
【反例】交接寫「大概率是 cache 層的問題，你從那裡查」——把自己的
錨定原樣傳染出去，fresh context 白開了。

## S4 — 外部服務／SaaS：vendor 選擇是花錢決策，user 拍板

**機制**：模型推薦訓練資料裡文件最漂亮的服務——恰好是 VC-funded、
free tier 設計來轉化付費的那批。模型對價格零信號，且訓練資料裡的
pricing 必定過時。

1. **引入外部服務 = 問 user**（本就對齊 kit-workflow 的 new-deps 大型
   檔位與 R3）。vendor 選擇與品味決策同類：**模型不拍板**。
2. **提案必附成本卡**：當下實查的 pricing（**禁止憑訓練記憶**——價格
   是 stale-training-data 重災區）、free tier 上限、用量 10x 時的月費、
   退出成本（資料能否匯出？換掉要不要重寫？）。
3. **鎖定測試**：「這服務明天漲價一倍，遷移路徑是什麼？」答案是
   「重寫」→ 不得無審批進場。
4. **預設棧 user 一次拍板**：boring／self-host-first 的預設棧寫進專案
   constraints，之後模型的工作是「套進這個棧」，不是「選 vendor」。

【正例】「全文搜尋建議：方案 A 自架 pg trgm（$0，夠用到 10 萬列）、
方案 B Algolia（實查 2026-07 定價：…，10x 用量 $X/月，退出=重寫查詢
層）。推薦 A，翻盤條件：需要 typo-tolerance。」【反例】直接
`npm i algolia` 並用「大家都用這個」帶過——三重違規：沒問、沒查價、
沒退出評估。

## S5 — 可驗證性架構（design for verifiability）

**機制**：驗證不對稱性——生成便宜、驗證貴；而驗證的成本是被架構在
**上游**決定的。邏輯埋在 UI 後面 = 每次驗證都要人眼點擊，又貴又不穩；
邏輯 headless 可驅動 = 驗證變成便宜的指令。S1–S4 每一招的成本都被
這條上游決策影響。

兩段式規則（防止它自己變成過度設計）：

1. **恆定生效（免費的分層紀律）**：業務邏輯不得**只能**透過 UI 觸達
   ——一律放在可直接呼叫的 function／module，UI 呼叫它、不含邏輯。
   設計時就這樣做成本≈0；貴的是事後改造（同 `ALTER ADD` 的不對稱）。
2. **痛點驅動（不預建）**：專用 CLI／API 驅動入口**不預建**；觸發
   條件：同一條流程**第二次**需要人工點 UI 才能驗證時，補一個
   （與 LESSONS 的「同一個坑第二次」是同一個模式）。
3. **豁免**：純展示層（外觀、動畫、文案）驗證終點本來就是人眼
   → 走 S1 截圖迴圈與 R4；造 CLI 驗證「好不好看」是類別錯誤。
4. **翻盤條件**：專案本質是 agent 密集回歸驗證的 → 門檻降為
   「第一次」——消費者必然出現時，預建不是投機是預知。

【正例】結帳金額計算放 `core/pricing.py`，UI 與測試各自呼叫；驗證
= 一條 pytest，不用開瀏覽器。【反例】折扣邏輯寫在 onClick handler
裡，每次驗證都要人工走完購物車流程——第二次人工點擊時就該重構了。

## 使用提示

- 與其他規則的分工：kit-judgment 管**通用**證據紀律（verified/
  unverified 二分、量測代替 hedge），本檔是它在五個高風險領域的
  **實例化**——訊號怎麼補、補到哪。兩者不衝突；S1/S4 的裁決規則
  就是 R4 的應用。
- 部署到專案後，建議把命中的條目（尤其 S1.3 設計禁令、S4.4 預設棧、
  S5.1 分層紀律）具體化成該專案 CLAUDE.md 的 constraints——本檔給
  的是通用形，constraints 給的是專案的實際值。
